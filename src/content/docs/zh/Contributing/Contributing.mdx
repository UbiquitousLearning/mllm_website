我们感谢所有给**mllm**贡献提交的开源贡献者,但也提请您注意一定的提交流程。我们欢迎各种各样的改进，包括但不限于纠正错误或排版错误、代码优化、引入新功能和性能优化。典型的贡献流程如下：
- （可选）了解**mllm**的基本知识。
- 阅读教程以克隆数据并运行**mllm**。
- 遵循[规定步骤](#contribution-steps)来实施您的修改。
- 以[指定格式](#submission-format)贡献代码。

## 贡献步骤

1. 您可以自行查找任务，例如[支持新模型](/zh/contributing/support_new_model), [支持新Tensor函数](/zh/contributing/add_new_tensor_function)或[添加新算子](/zh/contributing/add_new_operator)。
2. 符合开发标准，通过拉取请求（Pull Request，PR）提交您的代码，并确保通过GitHub持续集成（CI）测试。
3. 根据评审人员的反馈修订您的提交，直至合并。请注意，每次PR合并都需要获得相应子系统维护者的批准。解决问题时，请勿添加新提交。只需替换重设旧的提交，添加新的提交并强制推送您的原始补丁。PR应以rebase格式合并，避免使用中间的合并提交。
{/* 4. 如果您对子系统有任何疑问，可以在[此处]()找到相关维护人员。 */}

## 提交格式

1. 首先，提交标题的格式应如下：`type：content`
  - Type:
    - feat：新增功能。
    - fix：修复BUG。
    - style：格式（不影响代码执行的更改）。
    - refactor：重构（即不是添加新功能或修改错误的代码更改）。
    - perf：与优化相关，例如改进性能和体验。
    - chore：构建过程或辅助工具的更改。
  - Content：修改的简明描述；
    - 对于BUG修复，应为：修复......的BUG
    - 对于新功能，清楚地划分功能的属性。
    - 对于补丁问题，需要指定补丁的范围。补丁不能包含在一个大的提交中；保持原始提交的形状。
2. 提交消息必须详细描述此修改的内容
  - 对于与功能相关的提交：概述添加的功能或所做的具体修改。
  - 对于BUG修复提交：阐明BUG的来源以及解决方法。
3. 在提交消息的单独一行上添加问题链接，以便追踪。
4. 附加说明：
  - 请勿在提交消息中包含中文文本。
  - 如果使用英文冒号，请在后面添加一个空格。
  - 在git提交消息中不要忘记添加"Signed-off-by"标记。
5. PR提交示例：
    ```
      Title
      feat:add CPUConveluation1D operator

      Message
      A brief introduction to CPUConveluation1D...

      Usage:
      i = _Conveluation1D({i}, kernal, stride, bias, ...);

      https://github_issue_link

      Signed-off-by: 
    ```
